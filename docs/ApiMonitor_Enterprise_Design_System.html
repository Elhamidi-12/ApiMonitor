1. Introduction

The design system defines the visual language, UX rules, component standards, code conventions, and governance process for ApiMonitor.

#

Purpose of the Design System

This document is the main source of truth for how ApiMonitor should look, behave, and be implemented. It aligns product, UX, UI components, accessibility, architecture, and Blazor implementation practices.

Who should use it

Developers

Use it for component structure, CSS naming, Blazor patterns, validation behavior, code standards, and PR checklists.

Designers

Use it for layout, typography, spacing, interaction states, accessibility, and product consistency.

Stakeholders

Use it to understand quality standards, UX goals, and why implementation decisions are made.

How teams should work with it

  • Start every UI feature by checking the relevant component and UX pattern sections.
  • Use existing components before creating new ones.
  • Document a new pattern before it spreads across the codebase.
  • Reference this document in pull requests that affect UI, architecture, or shared code.
  • Update tokens, components, and guidance instead of creating one-off styles.

Principles and goals

Status first

Users must quickly know whether APIs are healthy, slow, down, disabled, or unknown.

Accessible by default

Keyboard, screen reader, focus, and contrast support are required.

Implementation ready

Guidance should translate directly into Razor, CSS, C#, and tests.

Enterprise maintainability

Patterns must scale across features, teams, and future requirements.

2. Product / Project Overview

ApiMonitor is a professional dashboard for monitoring API availability, latency, reliability, and operational health.

#

Application description

ApiMonitor allows users to register API endpoints, check them automatically, measure response time, store history, visualize health, and eventually alert teams when an API becomes slow or unavailable.

HealthyHealthy

2xx response under threshold.

SlowSlow

2xx response but above latency threshold.

DownDown

Timeout, exception, or HTTP 4xx/5xx.

UnknownUnknown

No check result exists yet.

DisabledDisabled

Monitoring is turned off.

Target users

UserNeedsPrimary screens
Backend developersQuickly see whether APIs are up, slow, or returning errors.Dashboard, API details, logs
DevOps / platform engineersMonitor reliability, investigate outages, configure alert rules.Dashboard, alerts, logs, settings
Team leadsUnderstand service health and reliability trends without reading raw logs.Dashboard, reports
AdministratorsManage endpoints, permissions, environments, and monitoring configuration.APIs, settings, users

Business goals

  • Reduce time to detect API failures.
  • Improve reliability visibility across services.
  • Create a strong enterprise Blazor portfolio application.
  • Provide a foundation for alerts, SLA tracking, environments, and reports.

User experience goals

  • Users understand system health in under 10 seconds.
  • Critical failures are visible without scrolling.
  • Forms are simple, validated, and forgiving.
  • Tables support scanning, filtering, and action discovery.
  • Empty states teach users what to do next.

3. Architecture

ApiMonitor should use layered architecture that separates domain rules, application use cases, infrastructure concerns, and Blazor UI.

#

High-level application architecture

ApiMonitor.Web
  • Blazor UI, Razor Components, layouts, pages, auth UI, SignalR UI behavior.
  • Depends on Application contracts and shared UI components.
ApiMonitor.Application
  • Use cases, services, DTOs, validation orchestration, interfaces.
  • Depends on Domain. Avoid direct EF Core dependencies in strict architecture.
ApiMonitor.Domain
  • Entities, enums, value objects, domain rules, domain events.
  • No dependency on Web, Infrastructure, EF Core, or external services.
ApiMonitor.Infrastructure
  • EF Core, HTTP checks, repositories, integrations, identity implementation, background workers.
  • Implements Application interfaces.

Frontend architecture

  • Use Razor Components for pages and reusable UI elements.
  • Keep pages thin: load data, coordinate actions, delegate logic to services.
  • Move repeated UI into Components/Shared.
  • Use layouts for shell, sidebar, topbar, and authorized sections.
  • Use design tokens and component classes instead of inline styles.

Backend architecture

  • Use Application services for business workflows.
  • Use Infrastructure for database access, HTTP monitoring, external notifications, and background jobs.
  • Use EF Core migrations for database schema changes.
  • Use hosted services for periodic monitoring workers.
  • Use structured logging for operational events.

API architecture

ApiMonitor can begin as a server-rendered Blazor application using direct service injection. If an external API layer is added, expose predictable REST endpoints.

Authentication and authorization approach

  • Use ASP.NET Core Identity or enterprise identity provider when login is needed.
  • Roles: Admin, Operator, Viewer.
  • Use policy-based authorization for delete, edit settings, manage users, and alert rules.
  • Hide unauthorized UI actions, but enforce authorization on the server.

  • Components/Pages: routable pages only.
  • Components/Shared: reusable visual and interaction components.
  • Components/Layout: app shell, nav menu, topbar, sidebars.
  • Application/DTOs: transfer models for UI and APIs.
  • Application/Abstractions: service and repository interfaces.

Dependency rules and separation of concerns

Scalability considerations

  • Background workers should use scoped services through IServiceScopeFactory.
  • Monitoring intervals must be configurable and should avoid overwhelming external APIs.
  • Use retention policies to avoid unbounded check history growth.
  • Use pagination and server-side filtering for logs and history.
  • Use SignalR for live status updates but keep database as source of truth.
  • Prepare for environments and tenants by keeping endpoints grouped and queryable.

Blazorise component usage rules

Do

  • Use Blazorise components for standard controls: Button, TextInput, Select, Check, Card, Modal, Alert, Badge, Table, DataGrid.
  • Use ApiMonitor wrappers for product-specific patterns.
  • Use the semantic color mapping in the style guide.
  • Keep custom CSS limited to tokens, layout, and documented overrides.

Do not

  • Do not mix Blazorise, raw Bootstrap, and custom components randomly on the same feature.
  • Do not override Blazorise internals with fragile selectors unless documented.
  • Do not create one-off colors, margins, shadows, or button styles.
  • Do not use raw HTML form controls when a Blazorise component exists, unless there is a specific reason.

5. Design Principles

These principles guide design reviews, implementation decisions, and tradeoffs.

#

Consistency

Use the same visual language, naming, spacing, and behavior across all screens.

Accessibility

Every feature must work with keyboard, screen readers, visible focus, and sufficient contrast.

Responsiveness

Design mobile-first, then enhance for tablet and desktop.

Reusability

Build reusable components for repeated patterns and avoid copy-pasting markup.

Performance

Minimize unnecessary renders, expensive queries, large DOM tables, and heavy JavaScript.

Maintainability

Prefer simple components, clear services, predictable folder structure, and strong names.

Simplicity

Use the simplest UI that solves the problem.

Internationalization readiness

Do not hard-code culture-specific date, time, or number formatting.

8. UX Patterns

Reusable interaction patterns reduce confusion and make the application predictable.

#

Form validation

  • Validate required fields after submit and optionally after blur.
  • Show validation summary at the top for forms with more than three fields.
  • Show field-level validation directly under the related input.
  • After failed submit, move focus to validation summary or first invalid field.
  • Validation messages must tell users how to fix the issue.
BadGood
Invalid inputEnter a valid URL, for example https://api.example.com/health.
Name requiredAPI name is required.
ErrorCheck interval must be between 5 and 300 seconds.

Error handling

  • Use inline errors for field validation.
  • Use page-level errors when the entire page cannot load.
  • Use alerts when a panel or operation fails.
  • Use toasts only for short confirmations.
  • Log detailed errors server-side; show friendly messages to users.

Confirmation flows

  • Deleting an API endpoint requires confirmation.
  • Disabling monitoring for production APIs should require confirmation.
  • Use clear destructive button labels like Delete API.
  • Keep Cancel as the safe default action.

Loading behavior

  • Show skeletons for dashboards and tables when layout is known.
  • Disable submit buttons while saving.
  • Show inline loading for small operations such as Test now.
  • If loading takes more than 5 seconds, show a helpful message.

9. Accessibility

Accessibility is a functional requirement. A feature is not complete until it is usable with keyboard and assistive technologies.

WCAG guidance

  • Target WCAG 2.2 AA for enterprise screens.
  • Normal text contrast should be at least 4.5:1.
  • Large text contrast should be at least 3:1.
  • Focus indicators must be visible and not rely on color alone.
  • Error identification and instructions must be textual.

Keyboard navigation

  • All interactive elements must be reachable by Tab.
  • Use Enter/Space for buttons and controls.
  • Use Escape to close modals, drawers, menus, and popovers.
  • Do not create keyboard traps except focus trapping inside modal dialogs.
  • Maintain logical tab order matching visual order.

  • Status colors must be paired with text labels.

public sealed class EfApiEndpointRepository(ApiMonitorDbContext db) : IApiEndpointRepository
{
    public Task<ApiEndpoint?> GetByIdAsync(Guid id, CancellationToken cancellationToken)
        => db.ApiEndpoints.FirstOrDefaultAsync(x => x.Id == id, cancellationToken);

    public async Task<IReadOnlyList<ApiEndpoint>> GetAllAsync(CancellationToken cancellationToken)
        => await db.ApiEndpoints.AsNoTracking().OrderBy(x => x.Name).ToListAsync(cancellationToken);

        => await db.ApiEndpoints.AddAsync(endpoint, cancellationToken);

    public Task SaveChangesAsync(CancellationToken cancellationToken)
        => db.SaveChangesAsync(cancellationToken);
}

16. Security Guidelines

Security rules apply to UI, services, monitoring, logging, data storage, and deployment.

#

Authentication

  • Use ASP.NET Core Identity or enterprise SSO when login is required.
  • Require HTTPS in production.
  • Use secure cookies with appropriate SameSite settings.
  • Use multi-factor authentication if the app controls production monitoring.

Authorization

  • Use roles and policies for sensitive actions.
  • Enforce authorization server-side, not only in UI.
  • Audit changes to endpoints, alert rules, and settings.

Input validation

  • Validate all form input server-side.
  • Only allow supported HTTP methods.
  • Validate URLs and consider restricting private network targets in production to prevent SSRF.
  • Set maximum lengths for text fields.

Output encoding

  • Razor encodes output by default; do not render untrusted HTML.
  • Never display raw response bodies as HTML.
  • Use safe JSON formatting for response previews.

CSRF/XSS considerations

  • Use built-in ASP.NET Core antiforgery protections for forms and APIs where applicable.
  • Avoid inline scripts in production app pages.
  • Do not store tokens in local storage unless strongly justified and mitigated.
  • Sanitize or encode any user-provided content.

Secure API usage

  • Use timeouts for all monitored API calls.
  • Do not follow unlimited redirects.
  • Do not allow arbitrary internal network scanning in production without controls.
  • Protect stored headers, bearer tokens, and API keys.

Secrets management

  • Use .NET user secrets locally.
  • Use environment variables or managed secret stores in deployment.
  • Never commit secrets to Git.
  • Rotate compromised tokens immediately.

Logging sensitive data

  • Never log authorization headers, API keys, cookies, passwords, or full connection strings.
  • Mask sensitive values in diagnostics.
  • Log endpoint IDs and hostnames rather than full secret-bearing URLs.

17. Performance Guidelines

Performance matters because monitoring dashboards update frequently and can contain large historical datasets.

#

Blazor performance best practices

  • Keep components small and avoid unnecessary re-rendering.
  • Use @key when rendering changing lists.
  • Use ShouldRender carefully for high-frequency components.
  • Batch SignalR updates when updates are frequent.
  • Dispose subscriptions and timers.

Lazy loading

  • Load dashboard summary first, then detailed charts.
  • Load logs and history only when the user opens those sections.
  • Use pagination or infinite loading for long histories.

Rendering optimization

  • Do not render thousands of table rows at once.
  • Use virtualization for large lists.
  • Use DTOs shaped for the UI.
  • Avoid complex calculations in Razor markup.

Asset optimization

  • Keep JavaScript minimal.
  • Minify CSS in production.
  • Cache static assets with versioning.
  • Avoid heavy icon or chart libraries unless needed.

Caching

  • Cache expensive dashboard summary queries briefly.
  • Do not cache sensitive user-specific data in shared caches without keys.
  • Use server memory cache carefully and invalidate when endpoints change.

API performance

  • Use timeouts for API monitoring calls.
  • Use IHttpClientFactory.
  • Limit concurrency for background checks.
  • Use efficient indexes on check history tables.

Large data handling

  • Use server-side pagination for logs.
  • Use aggregate tables or queries for long-range charts.
  • Apply retention policies for old check results.
  • Export large datasets asynchronously.

20. Final Developer Checklist

Every UI or architectural change should pass this checklist before it is merged.

#

UI checklist

  • Uses documented components, tokens, and am- classes.
  • Has loading, empty, error, and success states where applicable.
  • Uses clear page title, description, and primary action.
  • Uses semantic HTML and correct landmarks.
  • Works in light and dark mode.
  • Works on mobile, tablet, and desktop.
  • Does not rely only on color for status.
  • Uses accessible labels and validation messages.
  • Keyboard navigation works.
  • Focus states are visible.

Architecture checklist

  • Domain does not reference infrastructure or UI.
  • Application logic is not hidden inside Razor components.
  • Infrastructure implements interfaces and contains external dependencies.
  • Database queries are not duplicated across pages.
  • Services use async methods and cancellation tokens where appropriate.
  • Large datasets use pagination, filtering, or virtualization.

Security checklist

  • Server-side authorization protects sensitive actions.
  • User input is validated server-side.
  • Untrusted content is encoded and not rendered as raw HTML.
  • Secrets are not committed or logged.
  • Monitoring URLs and headers are handled safely.

Testing checklist

  • Unit tests cover core logic and validation.
  • Component tests cover reusable UI behavior.
  • Manual QA covers create, edit, delete, validation, and error states.
  • Accessibility checks are completed.
  • Critical workflows are tested in at least one browser.
Definition of done: a feature is complete only when it is usable, accessible, tested, documented when necessary, secure, and consistent with this design system.
